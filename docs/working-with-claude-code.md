# Claude Codeとの働き方ガイド（非エンジニア向け）

このプロジェクトには、Claude Codeが事故を起こしにくくする仕組みが組み込まれています（詳しくはREADMEの「安全網の範囲と限界」）。ただし、その仕組みが効くかどうかは**使う人の行動**にも懸かっています。このガイドは、エンジニアでない人がClaude Codeと安全に働くための最低限の約束をまとめたものです。

## 許可を求められたときの判断のしかた

Claude Codeは、影響の大きい操作をする前に「この操作を許可しますか？」と確認してきます。ここが人間の一番大事な出番です。

- **意味が分からない操作は、許可する前に質問してください。**「このコマンドは何をするもの？失敗したらどうなる？やり直せる？」と聞けば、平易な言葉で説明してくれます。説明を読んでから判断して遅くありません。
- **目安**: ファイルやコードを「読む」操作はほぼ安全です。「変更する・削除する・外部に送る（push、デプロイ、メール等）」操作は、何が起きるか理解してから許可してください。
- **「常に許可」は慎重に。** 一度「常に許可」にすると、以降は確認なしで同じ種類の操作が実行されます。毎回聞かれて煩わしい読み取り系だけにとどめるのが安全です。
- 「確認がうるさいから全部許可して」とは考えないでください。確認は安全網の一部です。

## セッション（会話）の分けかた

- **1つの作業につき1つのセッション**を基本にしてください。「ログイン機能を作る」と「先週のバグを直す」を同じ会話で続けると、文脈が混ざって精度が落ちます。
- 会話が長くなってきたら、キリの良いところで新しいセッションを始めてください。前の作業の内容は、必要ならCLAUDE.mdやコミット履歴に残っています。

## Claudeの報告の読みかた

作業が終わるとClaude Codeは報告を書きます。全部読む必要はありませんが、**次の2つだけは毎回確認してください**。

1. **「何をしたか」**: 頼んだことと違うことをしていないか。
2. **「テスト・確認の結果」**: 「テストが失敗しています」「確認できていません」と書いてあるのに、そのまま「OK、次へ」と返さないでください。失敗や未確認の報告は、直してもらってから先に進むためのサインです。

Claude Codeは正直に報告するよう設計されていますが、読み飛ばされると意味がありません。

## 頼みかたのコツ

- **手段ではなく目的を伝える。**「◯◯というコマンドを実行して」より「お客さんがログインできない問題を直したい」の方が、安全で適切な方法を選んでくれます。
- **決まり文句で安全な手順が発動します**:
  - 壊れた・おかしくなった → 「**壊れたので戻したい**」（安全な復旧手順が始まります。自分でgitコマンドを調べて実行しないでください）
  - 公開・リリースしたい → 「**公開したい**」（公開前の安全チェックが先に走ります）
  - 定期点検 → 「**健康診断して**」（放置されがちな更新・警告をまとめて棚卸しします。週1回がおすすめ）
  - この自動発動は**Claude Codeを使っているときだけ**効きます。CodexやCursorなど別のAIツールを使う場合は、「`.claude/skills/safe-rollback/SKILL.md` の手順に従って戻して」のように手順書のファイルを指定して頼んでください（守られる範囲もClaude Codeより狭くなります。READMEの「使うAIツールによる強制力の違い」参照）。

## やってはいけないこと

- 安全のためのブロックや警告を「回避する方法」を探し続けること。ブロックには理由があります。理由を聞いて、それでも必要なら「なぜ必要か」を伝えてください。
- `.claude/settings.json` の deny設定やhookを、意味を理解しないまま削除・緩和すること。
- エラーや警告の報告を読まずに承認し続けること。

## この仕組みの限界

Claude Codeのチェックは事故の確率を大きく下げますが、保証ではありません。**個人情報・決済・健康情報などを扱うものを一般公開する前には、人間の専門家のレビューを受けてください。** 詳しくはREADMEの「安全網の範囲と限界」を読んでください。
